# Грабли при портировании ESP32-S2 → ESP32-S3 + TinyUF2

## 1. Serial через USB CDC ломает автозапуск после прошивки

**Симптом:** после того как TinyUF2 пишет новую прошивку и делает `esp_restart()`, светодиод TinyUF2 остаётся зелёным (ready), приложение не стартует. Помогает только ручной RESET.

**Причина:** в момент `esp_restart()` USB-OTG (TinyUSB) внутри TinyUF2 ещё держит активное соединение с хостом. Bootloader делает soft reset периферии, но OTG-контроллер на S3 остаётся в грязном состоянии. Когда наше приложение с `CDCOnBoot=cdc` поднимает TinyUSB в `initArduino()` (ещё до `setup()`), захват OTG падает и USB-стек виснет. Это специфика S3 — на S2 такого не было.

**Решение:** в боевой версии Serial через USB не нужен (есть экран). Собираем с `CDCOnBoot=default` — USB автоматом не поднимается в `initArduino()`, никакого конфликта. Для dev-сборки можно вернуть Serial, но знать, что первый запуск после прошивки требует ручного RESET.

## 2. Софтверное переключение "нормальный режим ↔ MSC" через RTC-флаг

**Что пробовали:** один глобальный `USBMSC`, статический конструктор до `USB.begin()`, по RTC-флагу решать — поднимать MSC или нет. Команда в Serial ставит флаг и перезагружает.

**Симптом:** любая попытка инициализировать MSC в глобальной фазе приводила к "кирпичу" — ни Serial, ни диск. Менял `USBMode`, `CDCOnBoot`, порядок `begin()` — не лечилось.

**Причина:** порядок static init в C++ не определён, а на S3 к этому добавляется грязный OTG после TinyUF2 (см. проблема 1). Поднимать TinyUSB до `setup()` — лотерея.

**Решение:** никаких режимов. `setup()` ждёт ~1.5 сек (OTG отходит), потом вручную `USB.begin()` + `USBMSC.begin()` — и диск виден всегда, при любом подключении к ПК. Чтение файлов с FFat делается **до** MSC-монтирования, один раз в `setup()`. В версии с аккумулятором просто не будем отдавать файл контроллеру, пока USB подключён к хосту — это решается на уровне логики, а не USB-стека.

## 3. PDM TX на S3: почему 19 итераций до рабочего конфига

**Симптом:** `i2s_channel_init_pdm_tx_mode()` возвращает `ESP_ERR_INVALID_ARG` (`IARG`) на любом `sample_rate > 48 кГц`. На 48 кГц всё работает.

**Корневые ограничения драйвера** (из `esp_driver_i2s/i2s_pdm.c` и заголовков):
- `I2S_PDM_TX_CLK_DAC_DEFAULT_CONFIG(rate)` задаёт `fp=960, fs=rate/100, bclk_div=13`.
- В `i2s_pdm_tx_set_clock()` есть жёсткая проверка `up_sample_fs <= 480, иначе IARG`. Для `rate > 48000` дефолтный `fs = rate/100 > 480` → IARG **до того, как драйвер вообще посмотрит на clk_src**.
- `mclk = rate * 64 * (fp/fs) * max(bclk_div, 8)`, требуется `mclk_div = sclk / mclk >= 1`. Если `sclk` источника не хватает — тот же `IARG`.
- `bclk_div < 8` принудительно поднимается до 8 (`I2S_PDM_TX_BCLK_DIV_MIN`).
- `I2S_CLK_SRC_APLL` на S3 для I2S не пробрасывается. То есть нет "чистой" частоты для дробных rate.

**Почему намотали столько итераций:**
- Начинали с `I2S_PDM_TX_CLK_DAC_DEFAULT_CONFIG(384000)`, не зная про ограничение `fs <= 480`. Без override это **гарантированно** IARG для любого `rate > 48k`. Весь первый виток гипотез был мимо: драйвер падал ещё на валидации, а мы крутили `clk_src` и думали что виноват источник.
- Когда стали override-ить отдельные поля (`mclk_multiple=128`, ручной `up_sample_fs=480`, `bclk_div`), ломали "контракт" DAC default config: `Fpdm` уезжал, и комбинация переставала проходить уже по `mclk_div`.
- На `DEFAULT` (`PLL_F160M` = 160 МГц) для `rate=192k, fp/fs=2, bclk=8`: `mclk = 192000*1024 ≈ 196.6 МГц > 160 МГц` → `mclk_div=0` → IARG. То есть даже технически "правильный" конфиг на 160M не имел шансов, обязателен `PLL_240M`.
- Отдельно проверяли гипотезу про compound literal в C++ через if/else с временной переменной — могло портить инициализацию структуры; в итоге не подтвердилось, но времени съело.
- `arduino-cli` без `--clean` иногда брал старый объектник `tone_gen.cpp.o` — выглядело как "правка не применилась". Лечится `--clean` в `release.sh`.
- Версии `dev9/dev10/...` шли через перезапись `VERSION` и сборки случайно перетёрли друг друга → ложное ощущение что "сломалось то, что только что работало".
- Диагностический OLED висит на I²C; в какой-то момент SDA отвалился и экран замёрз на последнем валидном кадре. По этому кадру я принял несуществующий "успех `192k @ 240M`" и потом полсессии пытался "восстановить" режим, который никогда не был рабочим.
- Первый вариант `tryAll` стартовал реальный `writerTask` внутри каждой попытки и потом не корректно её гасил → черный экран, висящий канал, ВЧ "расчёска" на пине от первого же успешного `enable`.

**Что реально подтверждено как рабочее:**
- `rate=192000, clk_src=PLL_240M, fp=480, fs=480, bclk_div=8` — `OK`, но `fp/fs=1`, sigma-delta без oversampling/sinc, шум выше.
- `rate=96000, clk_src=PLL_240M, fp=960, fs=480, bclk_div=8` — `OK`, `fp/fs=2`, нормальный режим с интерполяцией. Принят как боевой.
- На `DEFAULT`/`PLL_F160M` для `rate >= 96k` с `fp/fs=2` — `IARG` всегда (`sclk` не хватает).

**Аппаратное ограничение, которое надо помнить:**
- В `I2S PDM TX` `sample_rate` **общий на оба канала** (`GPIO1`/`GPIO2`). Разные частоты на двух выходах в одном I2S-инстансе невозможны.

**Итог:** основной источник времени — `I2S_PDM_TX_CLK_DAC_DEFAULT_CONFIG` с автоматическим `fs = rate/100`, который ломает валидацию для всего, что выше 48 кГц, плюс невозможность взять `APLL`. Без override `fs` и без `PLL_240M` высокие `rate` не запускаются в принципе. Если бы это знали с самого начала — обошлись бы 2–3 итерациями.
