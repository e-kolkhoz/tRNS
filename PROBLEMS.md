# Грабли при портировании ESP32-S2 → ESP32-S3 + TinyUF2

## 1. Serial через USB CDC ломает автозапуск после прошивки

**Симптом:** после того как TinyUF2 пишет новую прошивку и делает `esp_restart()`, светодиод TinyUF2 остаётся зелёным (ready), приложение не стартует. Помогает только ручной RESET.

**Причина:** в момент `esp_restart()` USB-OTG (TinyUSB) внутри TinyUF2 ещё держит активное соединение с хостом. Bootloader делает soft reset периферии, но OTG-контроллер на S3 остаётся в грязном состоянии. Когда наше приложение с `CDCOnBoot=cdc` поднимает TinyUSB в `initArduino()` (ещё до `setup()`), захват OTG падает и USB-стек виснет. Это специфика S3 — на S2 такого не было.

**Решение:** в боевой версии Serial через USB не нужен (есть экран). Собираем с `CDCOnBoot=default` — USB автоматом не поднимается в `initArduino()`, никакого конфликта. Для dev-сборки можно вернуть Serial, но знать, что первый запуск после прошивки требует ручного RESET.

## 2. Софтверное переключение "нормальный режим ↔ MSC" через RTC-флаг

**Что пробовали:** один глобальный `USBMSC`, статический конструктор до `USB.begin()`, по RTC-флагу решать — поднимать MSC или нет. Команда в Serial ставит флаг и перезагружает.

**Симптом:** любая попытка инициализировать MSC в глобальной фазе приводила к "кирпичу" — ни Serial, ни диск. Менял `USBMode`, `CDCOnBoot`, порядок `begin()` — не лечилось.

**Причина:** порядок static init в C++ не определён, а на S3 к этому добавляется грязный OTG после TinyUF2 (см. проблема 1). Поднимать TinyUSB до `setup()` — лотерея.

**Решение:** никаких режимов. `setup()` ждёт ~1.5 сек (OTG отходит), потом вручную `USB.begin()` + `USBMSC.begin()` — и диск виден всегда, при любом подключении к ПК. Чтение файлов с FFat делается **до** MSC-монтирования, один раз в `setup()`. В версии с аккумулятором просто не будем отдавать файл контроллеру, пока USB подключён к хосту — это решается на уровне логики, а не USB-стека.
